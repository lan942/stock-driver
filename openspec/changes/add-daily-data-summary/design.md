## Context

当前项目已具备股票列表管理、实时行情爬取、日线数据爬取等功能，但缺少对数据库中每日数据覆盖情况的可视化查看能力。用户无法直观了解哪些日期的数据完整、哪些日期需要补充，导致数据补全工作缺乏目标性。

## Goals / Non-Goals

**Goals:**
- 提供后端 API 接口，按日期统计 StockDaily 表中的数据条数
- 提供前端页面，以日历热力图或表格形式展示每日数据覆盖情况
- 支持按日期范围筛选，快速定位缺失数据的日期
- 显示每日数据完整性百分比（当前条数 / 总股票数）

**Non-Goals:**
- 不修改数据库表结构
- 不涉及数据补全的自动触发逻辑
- 不添加新的爬取策略

## Decisions

### 1. API 设计

**决策:** 新增 `/stocks/daily_summary` GET 接口

**理由:** 采用 RESTful 风格，与现有接口保持一致。接口返回按日期分组的数据统计，包含日期、数据条数、覆盖率等字段。

**替代方案:** 
- 使用 `/crawler/stats` - 但该路径更适合爬取状态统计
- 使用 `/analytics/daily_summary` - 但当前项目中 analytics 主要用于技术分析指标

**最终方案:** 在 `api/routes.py` 中新增 `GET /stocks/daily_summary` 接口，支持 `start_date` 和 `end_date` 参数筛选。

### 2. 数据查询逻辑

**决策:** 在 `services/stock_service.py` 中新增 `get_daily_summary` 方法

**理由:** 遵循项目的三层架构（api→service→model），将业务逻辑封装在 service 层，便于测试和复用。

**实现方式:** 使用 SQLAlchemy 的 GROUP BY 查询，按日期统计 StockDaily 表记录数，并计算覆盖率。

### 3. 前端展示

**决策:** 使用 Element Plus 的表格组件展示每日数据统计

**理由:** 
- 项目已使用 Element Plus，保持技术栈一致性
- 表格形式直观展示每日数据条数和覆盖率，便于对比分析
- 支持排序和筛选功能，方便用户定位问题日期

**替代方案:**
- 日历热力图 - 视觉效果好但实现复杂度较高，且不如表格精确
- ECharts 图表 - 适合趋势分析，但不适合查看具体日期数据

**最终方案:** 新增 `src/views/DataSummary.vue` 页面，使用 el-table 组件展示数据。

### 4. 覆盖率计算

**决策:** 使用数据库中 Stock 表的记录数作为总股票数

**理由:** Stock 表记录了当前有效的股票列表，以此作为分母计算覆盖率能反映真实的数据完整性。

## Risks / Trade-offs

- **性能风险**: 如果数据量很大（多年日线数据），GROUP BY 查询可能较慢 → 缓解措施：添加日期范围筛选参数，默认只查询最近 90 天数据
- **覆盖率准确性**: 部分股票可能在特定日期不存在（如上市前、退市后），导致覆盖率无法达到 100% → 缓解措施：在前端标注说明，展示实际股票数量而非理论最大值
