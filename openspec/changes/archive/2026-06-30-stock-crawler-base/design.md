## Context

当前项目使用 akshare 获取 A 股数据，但缺乏容错机制。当接口触发限流时，爬取会直接失败，影响数据采集的连续性。需要在现有 `backend/services/crawler/` 基础上构建可靠的爬虫基础层。

现有代码架构：
```
backend/services/crawler/
├── crawler.py      # 现有简单爬虫，无重试机制
├── analysis.py     # 分析服务
```

## Goals / Non-Goals

**Goals:**
- 构建具有容错能力的爬虫基础层，支持限流自动处理
- 统一数据格式，消除不同接口的单位差异
- 提供可复用的爬虫基类，未来新增数据源时无需重复建设

**Non-Goals:**
- 不实现完整的任务调度系统（定时爬取后续迭代）
- 不实现前端界面（界面层后续迭代）
- 不处理登录类接口（需要 Cookie/Session 管理）

## Decisions

### Decision 1: 爬虫基类设计

采用策略模式 + 模板方法：
- `CrawlerBase`: 抽象基类，定义爬取流程骨架
- `RateLimiter`: 限流管理器，处理重试和切换逻辑
- 具体爬虫继承 `CrawlerBase`，只需实现数据获取方法

** Alternatives Considered:**
- 装饰器模式：缺点是重试逻辑分散，难以统一管理
- 外观模式：缺点是不够灵活，难以针对单个接口调参

### Decision 2: 限流处理策略

采用指数退避 + 备用接口切换：
1. 检测到限流（HTTP 429 或特定错误码）时，触发等待
2. 指数退避等待：1s → 2s → 4s → 8s，最大等待 60s
3. 超过最大重试次数后，切换到备用接口
4. 所有接口都失败时，记录错误并继续下一批

**Alternatives Considered:**
- 只重试不切换：简单但可靠性差
- 快速切换：用户体验好但可能加剧对方服务器压力

### Decision 3: 单位统一策略

在爬虫层统一转换，不在数据模型层处理：
- 所有价格单位统一为「元」
- 成交量单位统一为「股」
- 换手率统一为「%」
- 涨跌幅统一为「%」

数据源单位差异参考：
- akshare 东方财富接口：价格/成交量大部分已是标准单位
- 部分接口成交额为「万元」，需要转换
- 部分接口换手率为小数（如 0.05 表示 5%），需要转换

## Risks / Trade-offs

[Risk] 接口返回数据格式不稳定
→ [Mitigation] 添加数据验证，异常格式记录日志并跳过该字段，不中断整体流程

[Risk] 大量股票时爬取时间长
→ [Mitigation] 批量处理 + 进度回调，支持中断和恢复

[Risk] 备用接口数据字段不一致
→ [Mitigation] normalizer 层统一转换，确保输出格式一致

## Open Questions

2. 重试间隔的最大值是否需要可配置？（建议可配置）
