## Context

当前 Crawler.vue 页面有三个操作按钮：更新股票列表、更新实时行情、获取个股历史数据。点击后仅按钮显示 loading 状态，完成后在操作日志中显示一行文字结果。用户无法看到：
- 爬取进度（当前处理了多少只股票）
- 详细结果（成功数、失败数、耗时）
- 实时行情是否会因当日已有记录而被跳过

实时行情更新存在两个问题：
1. 手动触发时也会受 `has_today_success_record` 幂等检查影响（和调度器共用逻辑），导致用户点了按钮但可能实际没爬取
2. 东方财富快照接口在非交易时间返回的是最近一个交易日的数据，但 `price_date` 始终取当天日期，导致日期不准

## Goals / Non-Goals

**Goals:**
- 股票列表和实时行情更新操作显示进度条和结果摘要
- 实时行情更新支持强制刷新（跳过当日成功检查）
- 实时行情更新支持指定数据日期
- 后端 API 同步支持 `force` 和 `date` 参数
- 整体 UI 风格与现有 Element Plus 组件保持一致

**Non-Goals:**
- 不改变爬取核心逻辑（爬虫本身、数据标准化、数据库模型不变）
- 不实现 WebSocket 推送（用轮询替代）
- 不涉及个股历史数据功能的修改
- 不修改调度器的自动执行逻辑

## Decisions

### Decision: 进度展示采用"完成后显示总量"方案

**选择**: 爬取完成后一次性返回成功数、失败数、耗时，不做实时逐行进度。

**理由**:
- akshare 和直连 API 都是一次性返回全量数据，没有逐行回调机制
- 实时行情爬取耗时约 30-60 秒，用户能感知到在运行但不需要精确到每秒
- 实现简单，不引入后台任务队列

**替代方案**:
- 后台任务 + 轮询进度：需要引入任务状态表和异步执行，复杂度高
- WebSocket 推送：需要额外基础设施，当前项目不需要

### Decision: 强制刷新仅作用于手动触发的实时行情更新

**选择**: `POST /crawler/update_realtime?force=true` 时跳过 `has_today_success_record` 检查，始终执行爬取。调度器的自动执行逻辑不变。

**理由**:
- 用户手动点击就是明确想要更新，不应该被幂等检查挡住
- 调度器的自动任务仍需幂等性，避免重复爬取浪费资源
- 参数化实现，不影响原有逻辑路径

**替代方案**:
- 新建一个独立的"强制更新"API端点：代码重复，维护成本高

### Decision: 日期选择器默认值为当天

**选择**: 日期选择器默认选中当日，用户可手动修改。后端如果不传 `date` 参数则使用当日。

**理由**:
- 大多数场景下用户就是要当天的数据
- 非交易时间用户可以手动选择最近一个交易日
- 与东方财富快照 API 的行为一致（用户知道自己在做什么）

**替代方案**:
- 自动判断最近交易日：需要调用 `is_trading_day` 循环往前找，增加复杂度且可能误判
- 默认不传日期：后端默认为当天，和显式选中当天效果一样

### Decision: 进度与结果使用 Element Plus 组件

**选择**: 使用 `el-progress` 显示进度条，`el-result` 或 `el-descriptions` 显示结果摘要。

**理由**:
- 项目已使用 Element Plus，保持一致性
- 组件开箱即用，样式统一
- 比纯文字日志更直观

## Risks / Trade-offs

- **[风险] 爬取时间较长，用户可能关闭页面** → Mitigation: 后端爬取仍会完成并记录到 CrawlStatus，用户下次打开可在日志中看到
- **[风险] 强制刷新可能导致重复爬取** → Mitigation: 仅手动触发时可用，调度器自动任务仍保持幂等；且爬取是更新操作，重复执行只是覆盖数据
- **[风险] 用户选错日期导致数据日期标注错误** → Mitigation: 在日期选择器旁添加说明文字，提醒用户东方财富返回的是快照数据，日期应与实际交易日对应
- **[权衡] 无实时进度 vs 实现复杂度** → 接受没有精确进度，用 loading + 完成后结果展示替代，降低实现成本
