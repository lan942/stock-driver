## Context

当前系统已有股票列表、实时行情、日线数据、涨跌排行、数据爬取等功能，但缺少持仓管理模块。用户无法记录和追踪自己的股票持仓情况，这限制了后续买卖策略验证功能的开发。

项目采用三层架构（api → service → model），后端使用 Flask + SQLAlchemy + SQLite，前端使用 Vue 3 + Element Plus + ECharts。

## Goals / Non-Goals

**Goals:**
- 创建持仓模型（Portfolio）和交易记录模型（Transaction）
- 实现持仓管理相关的 REST API
- 创建持仓管理前端页面，包含持仓总览、持仓明细、交易记录
- 支持现金余额管理
- 实时计算持仓收益和收益率
- 作为后续买卖策略验证功能的基础

**Non-Goals:**
- 不支持多用户账户管理（当前仅单用户）
- 不涉及复杂的投资组合优化算法
- 不实现自动交易功能
- 不支持股票分红、送股等复杂操作（可后续扩展）

## Decisions

### Decision 1: 数据模型设计

**选择**: 创建 Portfolio 和 Transaction 两个独立模型，并使用 CashBalance 表存储现金余额

**理由**: 
- Portfolio 模型存储当前持仓状态，便于快速查询
- Transaction 模型记录所有交易历史，便于追溯和统计
- CashBalance 表单独存储现金余额，简化查询逻辑

**替代方案**: 仅使用 Transaction 模型，通过计算交易记录得出当前持仓。但此方案查询效率低，每次查询都需要计算历史交易。

### Decision 2: 利润计算方式

**选择**: 利润计算在服务层进行，结合 StockDaily 表的最新收盘价

**理由**: 
- StockDaily 表已有每日收盘价数据，可直接使用
- 在服务层计算利润，保持模型层简洁
- 实时利润基于最新收盘价，用户体验好

**替代方案**: 在模型层存储当前价格和利润。但此方案需要定时更新，增加复杂性。

### Decision 3: API 设计

**选择**: 使用 RESTful API，将持仓管理接口放在 `/api/portfolio/` 路径下

**理由**: 
- 符合项目现有的 API 设计风格
- 路径清晰，便于区分不同模块
- 支持标准的 CRUD 操作

### Decision 4: 前端组件设计

**选择**: 创建独立的 Portfolio.vue 页面组件，包含多个子功能模块

**理由**: 
- 保持与现有页面组件一致的结构
- 使用 Element Plus 的卡片、表格、对话框等组件，保持 UI 风格统一
- 组件化设计便于后续维护和扩展

## Risks / Trade-offs

### Risk 1: 最新价格获取可能延迟

**风险**: 如果 StockDaily 表没有最新日期的数据，利润计算可能不准确

**缓解措施**: 在服务层检查最新数据日期，如果不是今天则提示用户数据可能过期

### Risk 2: 股票代码验证

**风险**: 用户输入的股票代码可能不存在于 Stock 表中

**缓解措施**: 在添加持仓和交易时验证股票代码是否存在，不存在则返回错误

### Risk 3: 数据一致性

**风险**: Portfolio 和 Transaction 数据可能不一致（例如手动修改持仓但没有对应交易记录）

**缓解措施**: 当前版本允许手动修改持仓，后续可增加交易记录自动更新持仓的功能

### Risk 4: SQLite 并发访问

**风险**: SQLite 在高并发场景下可能出现锁问题

**缓解措施**: 当前项目为单用户使用，并发访问风险较低；后续如需多用户支持，可考虑更换数据库
